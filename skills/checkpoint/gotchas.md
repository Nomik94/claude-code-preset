# Checkpoint Gotchas

## 자주 발생하는 실수

### 1. 변경 후에 체크포인트 생성
❌ 리팩토링을 먼저 수행한 후 "체크포인트 만들어야겠다" 생각
→ ✅ 위험한 변경을 시작하기 **전에** 체크포인트 생성

변경 후 체크포인트는 이미 깨진 상태를 저장하는 것이다. 돌아갈 안전한 지점이 없다.

### 2. git stash로 작업 중인 내용 날림
❌ 체크포인트 생성을 위해 `git stash`를 사용하고 복원을 잊음
→ ✅ `git stash`보다 `git commit`으로 명시적 체크포인트 생성. WIP 커밋 사용

`git stash`는 스택 기반이라 다른 stash와 섞이면 작업 내용을 잃기 쉽다.

### 3. 체크포인트 없이 대규모 변경 시작
❌ "간단한 리팩토링이니까 체크포인트 안 만들어도 되겠지"
→ ✅ 리팩토링, 삭제, 마이그레이션, 구조 변경은 규모와 무관하게 체크포인트 필수

"간단할 것"이라는 판단이 틀렸을 때 되돌릴 방법이 없다.

### 4. 체크포인트 메시지 없이 생성
❌ `git commit -m "checkpoint"` — 무엇을 위한 체크포인트인지 알 수 없음
→ ✅ `git commit -m "checkpoint: [작업 설명] 전 안전 지점"` 형태로 목적 명시

여러 체크포인트가 쌓이면 어느 것으로 돌아가야 하는지 판단할 수 없다.

### 5. 여러 파일 변경을 하나의 체크포인트로 묶음
❌ DB 스키마 변경 + 서비스 로직 변경 + 테스트 변경을 한 번에 체크포인트
→ ✅ 논리적 단위별로 체크포인트 분리. 부분 롤백이 가능하도록

하나의 거대한 체크포인트는 "전부 되돌리기" 또는 "전부 유지"만 가능하다.
