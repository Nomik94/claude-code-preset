# Confidence Check Gotchas

## 자주 발생하는 실수

### 1. 신뢰도 점수 부풀리기
❌ 불확실한 부분이 있는데 90%+ 점수를 매기고 바로 구현 시작
→ ✅ 각 항목을 실제로 검증한 후에만 점수 부여. "아마 괜찮을 것" 금지.

확인 안 한 항목에 체크 표시하면 구현 중 되돌아와야 하는 비용이 더 크다.

### 2. 영향 범위 과소평가
❌ 변경할 파일만 보고 영향받는 파일을 파악하지 않음
→ ✅ import 관계, 테스트 파일, 설정 파일까지 영향 범위 추적

`grep`이나 참조 검색으로 해당 모듈을 사용하는 곳을 반드시 확인하라.

### 3. 기존 코드 파악 없이 평가
❌ 프로젝트 구조를 탐색하지 않고 "기존 코드 확인: 100%" 선언
→ ✅ 실제로 관련 디렉토리를 읽고, 기존 패턴/컨벤션을 파악한 후 점수 산정

특히 중복 구현 여부와 재사용 가능한 유틸/헬퍼 존재 여부를 반드시 확인하라.

### 4. 테스트 전략 누락
❌ 구현 방법만 생각하고 테스트 방법은 "나중에 생각"으로 미룸
→ ✅ 어떤 테스트를 작성할지, 어떤 fixture가 필요한지 구현 전에 결정

테스트 불가능한 설계를 구현한 후 뒤늦게 발견하면 전면 재설계가 필요하다.

### 5. Stack Detection 무시
❌ pyproject.toml이 있는데 FE 체크리스트만 확인하거나 그 반대
→ ✅ 프로젝트 루트에서 pyproject.toml / package.json 존재 여부를 먼저 확인

BE/FE/Fullstack 중 어떤 체크리스트를 적용할지 정확히 판단해야 한다.

### 6. 라이브러리 버전 미확인
❌ "사용할 라이브러리를 알고 있는가?" 항목에 이름만 알고 체크
→ ✅ pyproject.toml이나 package.json에서 실제 설치된 버전 확인 후 체크

프로젝트에 설치되지 않은 라이브러리를 사용하거나, 설치된 버전과 다른 API를 쓰면 빌드 실패한다.
