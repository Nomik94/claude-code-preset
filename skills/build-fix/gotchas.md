# Build Fix Gotchas

## 자주 발생하는 실수

### 1. 에러 메시지 안 읽고 추측으로 수정
❌ 에러 출력을 대충 보고 "아마 이거겠지"로 코드 수정
→ ✅ 에러 메시지의 파일명, 줄 번호, 에러 코드를 정확히 읽고 해당 위치 확인

추측 수정은 다른 곳을 고치거나 새로운 에러를 만든다.

### 2. 타입 에러를 `type: ignore`로 억제
❌ mypy 에러가 나면 `# type: ignore`를 붙여서 통과시킴
→ ✅ 실제 타입을 수정하거나 올바른 타입 어노테이션 추가

`type: ignore`는 타입 안전성을 포기하는 것이다. 런타임 에러로 돌아온다.

### 3. 린트 에러를 `noqa`로 무시
❌ ruff 에러에 `# noqa: XXXX`를 붙여서 통과시킴
→ ✅ 에러 원인을 이해하고 코드를 실제로 수정

`noqa`는 false positive인 경우에만 사용. 실제 문제를 숨기면 안 된다.

### 4. 한 에러 수정이 다른 에러를 유발하는 루프
❌ A를 고치면 B가 깨지고, B를 고치면 A가 깨지는 반복
→ ✅ 3회 실패 시 즉시 중단. 근본 원인(설계 문제)을 재분석

3+ Fix Rule: 같은 문제를 3번 수정 시도 실패하면 접근 방식 자체가 잘못된 것이다.

### 5. import 에러를 경로 수정으로만 해결
❌ `ModuleNotFoundError`가 나면 import 경로만 이리저리 바꿈
→ ✅ 패키지가 설치되어 있는지 (`poetry show`), `__init__.py`가 있는지 먼저 확인

의존성 미설치, `__init__.py` 누락, 순환 import가 진짜 원인인 경우가 많다.

### 6. 에러를 한 번에 하나씩만 수정
❌ 에러가 20개인데 하나 고치고 빌드 돌리고 반복
→ ✅ 같은 유형의 에러는 한 번에 일괄 수정 후 빌드

동일 패턴의 에러(예: import 순서, 타입 어노테이션 누락)는 일괄 처리가 효율적이다.
