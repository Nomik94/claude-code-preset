# Feature Planner Gotchas

## 자주 발생하는 실수

### 1. 스코프 크리프 — 요청하지 않은 기능 추가
❌ "사용자 생성 API" 요청에 비밀번호 재설정, 이메일 인증, 프로필 이미지까지 계획
→ ✅ 요청받은 기능만 계획. 추가 기능은 "향후 고려사항"으로 분리 표기

MVP 범위를 벗어나면 구현 시간이 기하급수적으로 늘어난다.

### 2. 의존성 순서 무시
❌ 서비스 레이어부터 구현하려는데 아직 모델/레포지토리가 없음
→ ✅ 의존성 그래프를 그리고 하위 레이어부터 순서대로 구현 계획

Model → Repository → Service → Controller 순서를 지켜야 각 단계에서 테스트 가능하다.

### 3. 테스트 계획 누락
❌ 기능 구현 파일만 나열하고 테스트 파일은 계획에 없음
→ ✅ 구현 파일마다 대응하는 테스트 파일과 테스트 케이스 목록 포함

테스트 없는 기능 계획은 불완전한 계획이다.

### 4. 기존 코드와 충돌 가능성 무시
❌ 새 기능만 생각하고 기존 코드에 미치는 영향을 분석하지 않음
→ ✅ 변경이 필요한 기존 파일 목록과 변경 이유를 명시

새 모델 추가 시 기존 migration, import, router 등록 등의 영향을 빠뜨리기 쉽다.

### 5. 파일 단위가 아닌 기능 단위로만 계획
❌ "사용자 CRUD 구현"처럼 추상적 태스크만 나열
→ ✅ 생성/수정할 파일 경로와 각 파일에 추가할 클래스/함수 명시

구체적이지 않은 계획은 구현 시 판단을 다시 해야 하므로 계획의 의미가 없다.

### 6. 마이그레이션 단계 누락
❌ 모델 변경을 계획하면서 Alembic 마이그레이션 생성 단계를 빠뜨림
→ ✅ DB 스키마 변경이 포함된 기능은 마이그레이션 파일 생성을 명시적 단계로 포함

모델만 바꾸고 마이그레이션을 안 만들면 런타임에서 테이블 불일치가 발생한다.
