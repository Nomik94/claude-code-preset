# 기능 플래너

코드 작성 전 기능 설계. 3개 이상 파일에 영향을 주거나 아키텍처 결정이 필요할 때 실행.

## Phase 기반 계획

### Phase 0: 스펙 문서 확인

구현 계획 수립 전 스펙 문서를 확인한다.

1. `docs/specs/constitution.md` 존재 여부 확인
2. `docs/specs/{feature-name}.spec.md` 존재 여부 확인
3. 스펙이 있으면:
   - Status가 LOCKED인지 확인 (DRAFT/REVIEW면 먼저 확정 요청)
   - 스펙의 사용자 시나리오, 입출력 정의, 에러 케이스를 Phase 1 입력으로 사용
   - Constitution 원칙을 Phase 4 품질 게이트에 반영
4. 스펙이 없으면:
   - 3개 이상 파일 변경 또는 새 기능이면 `/spec`으로 스펙 먼저 작성 **권장**
   - 기존 기능 확장이면 Phase 1부터 진행 가능

### Phase 1: 요구사항 명확화

네 가지 질문에 답변 (스펙 문서가 있으면 해당 섹션에서 추출):

1. **무엇**: 기능에 대한 한 문장 설명 ← spec §1 What
2. **범위 밖**: 이 기능이 하지 않는 것을 명시적으로 나열 ← spec §5 OUT
3. **완료 기준**: 구체적이고 테스트 가능한 조건 ← spec §3 성공 기준
4. **제약 조건**: 기술 스택 제한, 성능 요구사항, 데드라인 ← spec §6 비기능 요구사항

답할 수 없는 질문이 있으면 사용자에게 확인.

### Phase 2: 기존 코드 분석

설계 전 코드베이스 조사:

| 분석 항목 | BE (pyproject.toml) | FE (package.json) |
|----------|--------------------|--------------------|
| 관련 코드 | 모듈, 서비스, 모델 | 컴포넌트, 훅, 스토어 |
| 패턴 | 네이밍, 에러 핸들링, 응답 형식 | 컴포넌트 구조, 상태관리, 데이터 페칭 |
| 재사용 | 기존 서비스/유틸 확장 | 기존 컴포넌트/훅 확장 |
| 영향 범위 | 읽기/수정/생성 파일 전체 나열 | 읽기/수정/생성 파일 전체 나열 |

### Phase 3: 구현 계획 + 의존성 매핑

순서가 있는 작업 목록 작성:

1. 각 작업에 순번 부여
2. **병렬** 실행 가능한 작업과 **순차** 실행해야 하는 작업 표시
3. 각 작업별: 영향받는 파일, 예상 복잡도 (S/M/L)
4. **의존성 매핑**: 작업 간 선후 관계 명시
5. **위험 요소** 표시: 마이그레이션, 공유 유틸리티, 핵심 추상화
6. 10개 이상 파일에 영향을 주면 독립 배포 가능한 단계로 분할

### Phase 4: 품질 게이트

각 단계 완료 시 통과해야 하는 게이트:

| 게이트 | 기준 |
|--------|------|
| 타입 안전 | BE: mypy --strict 통과 / FE: tsc --noEmit 통과 |
| 테스트 | 신규 코드 커버리지 80% 이상 |
| 린트 | BE: ruff / FE: eslint 통과 |
| 아키텍처 | 레이어 위반 없음 |

### Phase 5: 범위 확정

구현 시작 전:
- **범위 내**와 **범위 밖** 목록을 명시적으로 출력
- 사용자 확인: "이 계획은 X를 포함합니다. 진행할까요?"
- 범위 확정 전 코딩 시작 금지

## 규칙

- 과잉 구현 금지: 요청받은 것만 빌드
- MVP 경로 우선: 가장 단순한 작동 버전 먼저
- 불확실한 점은 즉시 표면화
- 10개 이상 파일 변경 = 단계별 접근 필수
- 각 단계는 독립적으로 테스트 및 배포 가능

## 출력 형식

```
## Feature: [name]

### Requirements
- What: ...
- NOT in scope: ...
- Acceptance criteria: ...
- Constraints: ...

### Affected Code
| File | Action | Stack | Complexity |
|------|--------|-------|------------|
| app/services/user.py | modify | BE | S |
| src/components/UserForm.tsx | create | FE | M |

### Implementation Plan
| # | Task | Files | Depends On | Parallel | Risk |
|---|------|-------|------------|----------|------|
| 1 | DB 모델 추가 | 2 | - | no | low |
| 2 | API 엔드포인트 | 3 | 1 | no | low |
| 3 | FE 컴포넌트 | 4 | 2 | yes | med |

### Quality Gates
| Phase | Gate | Criteria |
|-------|------|----------|
| 1 완료 | Type | mypy + tsc pass |
| 2 완료 | Test | 80%+ coverage |

### Scope Lock
- IN: ...
- OUT: ...
```

자주 발생하는 실수는 이 디렉토리의 gotchas.md를 참조하라.
