# Verify Gotchas

## 자주 발생하는 실수

### 1. 실제 명령 실행 없이 "should pass" 선언
❌ `poetry run pytest`를 실행하지 않고 "테스트가 통과할 것입니다"라고 선언
→ ✅ 반드시 실제 명령어를 실행하고 전체 출력을 확인한 후에만 통과 선언

"should pass", "probably works", "looks correct"는 검증이 아니다.

### 2. Lint만 돌리고 Type Check 스킵
❌ `ruff check`만 통과하면 끝이라고 판단. mypy를 실행하지 않음
→ ✅ Lint(1단계)와 Type Check(2단계)는 별개. 둘 다 반드시 실행

ruff는 타입 에러를 잡지 못한다. `mypy --strict`에서 잡히는 에러는 별도로 존재한다.

### 3. 테스트 0개인데 통과 선언
❌ pytest 출력이 "0 tests collected"인데 "테스트 통과"로 보고
→ ✅ 수집된 테스트가 0개면 실패로 처리. 새 코드에는 반드시 테스트가 있어야 함

"0 passed"는 "all passed"가 아니라 "검증 없음"이다.

### 4. 일부 단계만 실행하고 전체 통과 선언
❌ 7단계 중 3단계만 실행하고 "검증 완료"
→ ✅ 7단계 모두 순서대로 실행. 하나라도 건너뛰면 "검증 미완료"

특히 Security(5단계)와 Dependencies(6단계)를 빠뜨리기 쉽다.

### 5. 경고(WARN)를 무시
❌ 파일 크기 300줄 초과 경고, TODO 잔존 경고를 무시하고 통과 처리
→ ✅ 경고는 보고서에 명시. 사용자에게 알리고 판단을 맡김

경고를 숨기면 기술 부채가 쌓인다.

### 6. 이전 실행 결과를 재사용
❌ 코드 수정 후 이전에 실행한 검증 결과를 그대로 인용
→ ✅ 코드가 변경되면 검증을 처음부터 다시 실행

캐시된 결과는 현재 코드 상태를 반영하지 않는다.
