# Production Checklist Gotchas

## 자주 발생하는 실수

### 1. Health check가 항상 200 반환
❌ `/health` 엔드포인트가 DB 연결, Redis 연결 등 실제 상태를 확인하지 않고 항상 200 반환
→ ✅ DB ping, Redis ping, 핵심 의존성 연결 상태를 실제로 확인한 후 응답

"항상 200"인 health check는 DB가 죽어도 "정상"으로 보고하여 장애 감지를 못한다.

### 2. 로그에 PII(개인식별정보) 포함
❌ 요청/응답 로깅에 이메일, 전화번호, 비밀번호, 주민번호 등 포함
→ ✅ PII 필드 마스킹. structlog processor에서 민감 필드 자동 필터링

PII 로그는 GDPR/개인정보보호법 위반이며, 로그 시스템 접근자에게 개인정보가 노출된다.

### 3. 롤백 절차 미준비
❌ 배포만 준비하고 문제 발생 시 이전 버전으로 돌아가는 절차가 없음
→ ✅ 배포 전에 롤백 명령어, 롤백 대상 버전, DB 마이그레이션 롤백 절차 문서화

"배포 후 장애 → 롤백 방법 모름 → 장시간 서비스 중단"은 가장 흔한 프로덕션 사고 패턴이다.

### 4. 에러 알림 미설정
❌ 500 에러가 발생해도 팀에 알림이 가지 않음 → 사용자 신고로 발견
→ ✅ 5xx 에러 비율, 응답 시간 임계값 초과 시 자동 알림 (Slack, PagerDuty 등)

모니터링 없는 프로덕션은 장님 운전이다. 에러를 사용자보다 먼저 발견해야 한다.

### 5. 환경 변수 기본값이 프로덕션에 부적합
❌ `JWT_SECRET = "CHANGE-ME"`, `DEBUG = True` — 기본값이 개발용
→ ✅ 프로덕션 환경에서 기본값 사용 시 시작 자체를 거부하는 검증 추가

`model_validator`에서 `env == "prod"`일 때 위험한 기본값을 차단하라.

### 6. 로그 레벨 prod에서 DEBUG
❌ 프로덕션 로그가 DEBUG 레벨 → 디스크/네트워크 비용 폭증 + 성능 저하
→ ✅ prod는 INFO 이상. 일시적 디버깅 필요 시 동적 로그 레벨 변경 지원

DEBUG 로그는 요청당 수십 줄을 출력하여 로그 스토리지 비용을 수십 배 늘린다.

### 7. graceful shutdown 미구현
❌ SIGTERM 수신 시 진행 중인 요청을 즉시 끊음
→ ✅ SIGTERM 핸들러에서 새 요청 거부 + 기존 요청 완료 대기 + 타임아웃 후 종료

K8s 롤링 업데이트 시 graceful shutdown 없이는 진행 중인 요청이 502로 실패한다.
