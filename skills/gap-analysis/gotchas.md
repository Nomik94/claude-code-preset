# Gap Analysis Gotchas

## 자주 발생하는 실수

### 1. 자기가 만든 코드를 자기가 리뷰
❌ 방금 작성한 코드를 바로 gap analysis 수행 → 편향된 결과
→ ✅ 요구사항 문서/스펙을 기준으로 체크리스트를 먼저 작성한 후, 코드와 대조

자기 코드는 "의도대로 작성됨"이라고 착각하기 쉽다. 스펙 기반 검증이 필수다.

### 2. Excess 체크 안 함 (Missing만 확인)
❌ "요구사항에 있는데 코드에 없는 것"만 찾고, "코드에 있는데 요구사항에 없는 것"은 무시
→ ✅ Missing과 Excess를 모두 검사. 요청하지 않은 기능도 gap이다.

Excess 코드는 유지보수 비용을 늘리고, 예상치 못한 사이드 이펙트를 만든다.

### 3. 표면적 비교만 수행
❌ "파일이 존재하는가?"만 확인하고 내용이 요구사항을 충족하는지 검증 안 함
→ ✅ 파일 존재 여부 + 함수/클래스 구현 내용 + 엣지 케이스 처리까지 확인

빈 함수(`pass`)나 TODO 주석이 있는 코드는 "구현됨"이 아니다.

### 4. 비기능 요구사항 누락
❌ 기능 요구사항만 체크하고 성능, 보안, 에러 핸들링 gap 무시
→ ✅ 기능 + 비기능(성능, 보안, 로깅, 에러 처리) 모두 gap 분석 대상

rate limiting, 입력 검증, 로그 출력 등 비기능 요구사항이 빠지면 프로덕션에서 문제 발생.

### 5. 테스트 커버리지를 gap으로 포함하지 않음
❌ 구현 코드만 gap 분석하고 테스트 코드는 범위 밖으로 취급
→ ✅ 각 요구사항에 대응하는 테스트가 있는지도 gap 항목에 포함

테스트 없는 구현은 "검증되지 않은 구현"이므로 gap이다.
