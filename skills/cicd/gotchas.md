# CI/CD Gotchas

## 자주 발생하는 실수

### 1. 캐시 키 잘못 설정
❌ 캐시 키에 lock 파일 해시를 포함하지 않음 → 의존성 변경 시에도 오래된 캐시 사용
→ ✅ `hashFiles('**/poetry.lock')` 또는 `hashFiles('**/pnpm-lock.yaml')`을 캐시 키에 포함

잘못된 캐시는 "로컬에서는 되는데 CI에서 안 됨" 문제의 흔한 원인이다.

### 2. 시크릿이 로그에 노출
❌ `echo $DB_PASSWORD` 또는 `env` 명령으로 시크릿이 CI 로그에 출력
→ ✅ 시크릿은 `${{ secrets.XXX }}`로만 참조. `--quiet` 플래그 사용. 로그 마스킹 확인

CI 로그는 팀원 전체가 볼 수 있다. 시크릿이 한 번 로그에 찍히면 교체해야 한다.

### 3. 병렬 실행 시 테스트 격리 안 됨
❌ 테스트가 같은 DB/포트/파일을 공유하여 병렬 실행 시 랜덤 실패
→ ✅ 테스트별 고유 DB 이름, 랜덤 포트, 임시 디렉토리 사용

CI에서 "가끔 실패하는 테스트"는 대부분 리소스 격리 문제다.

### 4. main 브랜치에 직접 push 허용
❌ CI가 PR 없이 main에 직접 push하는 것을 차단하지 않음
→ ✅ branch protection rule 설정: PR 필수 + CI 통과 필수 + 리뷰 승인 필수

보호 없는 main 브랜치는 깨진 코드가 바로 배포될 수 있다.

### 5. CI 환경과 로컬 환경 불일치
❌ CI는 Ubuntu + Python 3.12인데 로컬은 macOS + Python 3.13
→ ✅ CI 환경을 프로젝트 요구사항과 일치시키고, Docker로 로컬 재현 가능하게 구성

환경 차이로 인한 "CI에서만 실패"는 디버깅 비용이 매우 높다.

### 6. 실패한 step 이후에도 배포 진행
❌ 테스트 실패했는데 배포 step이 `if: always()`로 설정되어 그대로 배포
→ ✅ 배포 step은 모든 이전 step 성공 시에만 실행. `needs:` 의존성 명확히 설정

`if: always()`는 알림/정리 작업에만 사용. 배포에 사용하면 깨진 코드가 프로덕션에 올라간다.

### 7. 타임아웃 미설정
❌ 무한 루프나 데드락으로 CI 작업이 영원히 실행
→ ✅ job 레벨과 step 레벨에 `timeout-minutes` 설정

타임아웃 없는 작업은 CI 러너를 점유하여 다른 작업을 블로킹한다.
