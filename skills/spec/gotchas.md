# Spec Gotchas

## 자주 발생하는 실수

### 1. How를 Specify 단계에서 작성
❌ 스펙 문서에 "FastAPI로 구현", "React Query 사용" 등 기술 결정 포함
→ ✅ Specify는 What/Why만. How는 architect/feature-planner 단계에서 결정

기술 결정을 너무 일찍 하면 더 나은 설계 옵션을 차단한다.

### 2. Constitution 없이 스펙 작성
❌ 프로젝트 원칙이 없는 상태에서 개별 기능 스펙만 작성
→ ✅ 첫 스펙 작성 전 constitution.md 존재 여부 확인. 없으면 먼저 생성

기능별 품질/보안 기준이 일관되지 않으면 프로젝트 전체가 파편화된다.

### 3. 해피 패스만 명세
❌ 정상 동작 시나리오만 작성하고 에러/엣지 케이스 누락
→ ✅ "~하면 어떻게 되나?" 질문을 최소 5가지 던진 후 에러 케이스 섹션 완성

에러 케이스가 없는 스펙은 구현 시 개발자(또는 AI)가 임의로 결정하게 된다.

### 4. LOCKED 전에 구현 시작
❌ DRAFT 상태의 스펙을 기반으로 코드 작성 시작
→ ✅ 반드시 Status: LOCKED 확인 후 구현. 스펙 변경 시 구현도 변경해야 하므로 비용이 크다

미확정 스펙 기반 구현은 재작업의 주범이다.

### 5. CLAUDE.md 규칙을 Constitution에 중복
❌ "snake_case 사용", "Poetry 필수" 등 이미 CLAUDE.md에 있는 규칙을 constitution.md에 반복
→ ✅ Constitution은 프로젝트 **고유** 원칙만. 스택/컨벤션은 CLAUDE.md가 관장

중복은 불일치를 만들고, 불일치는 혼란을 만든다.

### 6. 스코프 없이 잠금
❌ IN/OUT 스코프 없이 "기능 명세 완료"로 LOCKED 처리
→ ✅ IN과 OUT을 모두 명시적으로 나열. 특히 OUT("이것은 안 한다")이 중요

스코프 없는 스펙은 끝없는 스코프 크리프의 시작이다.

### 7. Constitution을 한 번 쓰고 방치
❌ 프로젝트 초기에 작성 후 업데이트 없이 형식적 문서로 남김
→ ✅ 새로운 기능/이슈에서 원칙 부족을 발견하면 Constitution에 추가

살아있는 문서만 가치가 있다.
